Delete every ticket (any status)

Removes 100% of tickets in the database — same command for open, closed, resolved, pending approval, waiting on customer, and every other status. Related data (replies, approvals, audit) is removed with each ticket. Stored files under each ticket ID folder are deleted. Staff users and portal settings are not deleted. Public ticket sequence is reset to 999 (next ticket continues your normal numbering rules).

Type exactly to confirm:

DELETE_ALL_TICKETS
Purge all application data (full reset)

Deletes every ticket and related rows, resets portal settings to installation defaults (categories, branding, SLA, ticket ID rules, etc.), and clears ticket upload folders. Use the option above if you only want tickets removed while keeping portal configuration.

Type exactly to confirm:

PURGE_ALL_DATABASE_DATA

To remove accounts, type exactly:

DELETE_ALL_USERS
Ticket import & export (CSV)

CSV columns match the template and export. project is stored on each ticket (for grouping or reporting). ticket_number: leave blank to auto-assign the next public ID from portal rules; or set a unique value (up to 50 chars) to keep a legacy / external ticket id. No email is sent on import. For open statuses, leave resolved_date empty. UTF-8 CSV (Excel: CSV UTF-8).

    Auto Assignment Rules

    Choose how public tickets get an owner: off, round-robin A–Z, or custom queues. Set a per-priority order of agents and managers (1st, 2nd, …) with an on/off switch for each SLA priority. Staff or manager–created tickets assign to whoever created them. Priorities come from Ticket Categories & SLA Policies.

    Save applies assignment only. Ticket ID format and SLA priorities are edited under Ticket Categories & SLA Policies.

    Default mode & global queue
    Global custom order (when mode is Custom order): 1st receives the next matching ticket, then 2nd, … Remaining agents and managers follow alphabetically.
      Per-priority order (agents & managers)

      Enable a priority and add agents or managers in 1st → 2nd → … order. Other active agents and managers are appended A–Z. When off for that priority, the global rule above applies. Staff/manager–created tickets assign to the creator (no team-wide new-ticket email); the customer still gets the assignment email when the ticket has an owner.

      Enterprise notifications

      Email templates

      Standard templates for staff alerts, customer updates, manager approvals, and lifecycle events. Requires SMTP credentials in .env (MAIL_USER, MAIL_PASSWORD). Use {placeholders} in subject and body — each card lists typical fields.

      Common placeholders: {ticket_id} {subject} {ticket_url} (customer portal; staff/manager templates may also use {customer_portal_url} for the same link) {brand_name} {logo_url} {company_tagline} {customer_name} {customer_email} — see per-template descriptions.

      Email branding (HTML header)

      Shown at the top of HTML notification emails (staff, customer, manager approval). Plain-text bodies are unchanged. Logo must be a full https:// or http:// image URL.

      Ticket mail (SMTP) still authenticates with MAIL_USER / MAIL_PASSWORD in .env. Optional fields below set the visible From name/address and default Reply-To on ticket-related messages (assignment, replies, closed, forwards). Your provider must allow the From address as an alias or send identity.

      Leave blank to use MAIL_FROM from .env.
      Leave blank to use MAIL_FROM_NAME from .env.
      Used when the template does not set a more specific Reply-To (e.g. ticket forwards still reply to the staff member who forwarded).
